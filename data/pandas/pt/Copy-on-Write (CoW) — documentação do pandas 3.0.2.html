Copiar e Escrever (CoW)#

Nota

Copiar ao escrever agora é o padrão com o pandas 3.0.

O Copy-on-Write foi introduzido pela primeira vez na versão 1.5.0. A partir da versão 2.0, a maioria dos otimizações que se tornam possíveis por meio do CoW são implementadas e suportadas. Tudo possível Otimizações são suportadas a partir do Pandas 2.1.

O CoW levará a comportamentos mais previsíveis, já que não é possível atualizar mais do que Um objeto com uma única instrução, por exemplo, operações ou métodos de indexação, não terá efeitos colaterais. Além disso, através de Atrasando cópias o máximo possível, o desempenho médio e o uso de memória melhorarão.

Comportamento anterior#

O comportamento de indexação do Pandas é difícil de entender. Algumas operações retornam vistas enquanto outras cópias de retorno. Dependendo do resultado da operação, mutando um objeto pode acidentalmente mutar outro:

In [1]: df = pd.DataFrame({"foo": [1, 2, 3], "bar": [4, 5, 6]})
In [2]: subset = df["foo"]
In [3]: subset.iloc[0] = 100
In [4]: df
Out[4]:
   foo  bar
0  100    4
1    2    5
2    3    6

Mutando , por exemplo, atualizando seus valores, também atualizado . O comportamento exato era Difícil de prever. Copy-on-Write resolve modificar acidentalmente mais de um objeto, Isso é explicitamente proibido. permanece inalterado:subsetdfdf

In [1]: df = pd.DataFrame({"foo": [1, 2, 3], "bar": [4, 5, 6]})

In [2]: subset = df["foo"]

In [3]: subset.iloc[0] = 100

In [4]: df
Out[4]: 
   foo  bar
0    1    4
1    2    5
2    3    6

As seções a seguir explicarão o que isso significa e como impacta o que já existe aplicações.

Migrando para Copy-on-Write#

Copiar ao escrever é o modo padrão e único no pandas 3.0. Isso significa que os usuários precisam migrar o código deles para estar em conformidade com as regras do CoW.

O modo padrão no pandas < 3.0 gera avisos para certos casos que vão ativamente mudar o comportamento e, assim, mudar o comportamento pretendido pelo usuário.

Pandas 2.2 tem um modo de aviso

pd.options.mode.copy_on_write = "warn"

que alertará para cada operação que mudará o comportamento com a CoW. Esperamos esse modo Ser muito barulhento, já que muitos casos que não esperamos que eles influenciem os usuários vão Também emita um aviso. Recomendamos verificar esse modo e analisar os avisos, mas ele é Não é necessário abordar todos esses avisos. Os dois primeiros itens das listas a seguir são os únicos casos que precisam ser tratados para que o código existente funcione com o CoW.

The following few items describe the user visible changes:

Chained assignment will never work

loc should be used as an alternative. Check the chained assignment section for more details.

Accessing the underlying array of a pandas object will return a read-only view

In [5]: ser = pd.Series([1, 2, 3])

In [6]: ser.to_numpy()
Out[6]: array([1, 2, 3])

This example returns a NumPy array that is a view of the Series object. This view can be modified and thus also modify the pandas object. This is not compliant with CoW rules. The returned array is set to non-writeable to protect against this behavior. Creating a copy of this array allows modification. You can also make the array writeable again if you don’t care about the pandas object anymore.

See the section about read-only NumPy arrays for more details.

Only one pandas object is updated at once

The following code snippet updated both and without CoW:dfsubset

In [1]: df = pd.DataFrame({"foo": [1, 2, 3], "bar": [4, 5, 6]})
In [2]: subset = df["foo"]
In [3]: subset.iloc[0] = 100
In [4]: df
Out[4]:
   foo  bar
0  100    4
1    2    5
2    3    6

This is not possible anymore with CoW, since the CoW rules explicitly forbid this. This includes updating a single column as a Series and relying on the change propagating back to the parent DataFrame. This statement can be rewritten into a single statement with or if this behavior is necessary. DataFrame.where() is another suitable alternative for this case.lociloc

Updating a column selected from a DataFrame with an inplace method will also not work anymore.

In [7]: df = pd.DataFrame({"foo": [1, 2, 3], "bar": [4, 5, 6]})

In [8]: df["foo"].replace(1, 5, inplace=True)
Out[8]: 
0    5
1    2
2    3

In [9]: df
Out[9]:

1    2    5
2    3    6

This is another form of chained assignment. This can generally be rewritten in 2 different forms:

In [10]: df = pd.DataFrame({"foo": [1, 2, 3], "bar": [4, 5, 6]})

In [11]: df.replace({"foo": {1: 5}}, inplace=True)
Out[11]:

A different alternative would be to not use :inplace

In [13]: df = pd.DataFrame({"foo": [1, 2, 3], "bar": [4, 5, 6]})

In [14]: df["foo"] = df["foo"].replace(1, 5)

In [15]: df

Constructors now copy NumPy arrays by default

The Series and DataFrame constructors now copies a NumPy array by default when not otherwise specified. This was changed to avoid mutating a pandas object when the NumPy array is changed inplace outside of pandas. You can set to avoid this copy.copy=False

Description#

CoW means that any DataFrame or Series derived from another in any way always behaves as a copy. As a consequence, we can only change the values of an object through modifying the object itself. CoW disallows updating a DataFrame or a Series that shares data with another DataFrame or Series object inplace.

This avoids side-effects when modifying values and hence, most methods can avoid actually copying the data and only trigger a copy when necessary.

The following example will operate inplace:

In [16]: df = pd.DataFrame({"foo": [1, 2, 3], "bar": [4, 5, 6]})

In [17]: df.iloc[0, 0] = 100

In [18]: df
Out[18]: 
   foo  bar
0  100    4
1    2    5
2    3    6

The object does not share any data with any other object and hence no copy is triggered when updating the values. In contrast, the following operation triggers a copy of the data under CoW:df

In [19]: df = pd.DataFrame({"foo": [1, 2, 3], "bar": [4, 5, 6]})

In [20]: df2 = df.reset_index(drop=True)

In [21]: df2.iloc[0, 0] = 100

In [22]: df
Out[22]: 
   foo  bar
0    1    4
1    2    5
2    3    6

In [23]: df2
Out[23]: 
   foo  bar
0  100    4
1    2    5
2    3    6

reset_index returns a lazy copy with CoW while it copies the data without CoW. Since both objects, and share the same data, a copy is triggered when modifying . The object still has the same values as initially while was modified.dfdf2df2dfdf2

If the object isn’t needed anymore after performing the operation, you can emulate an inplace-like operation through assigning the output of to the same variable:dfreset_indexreset_index

In [24]: df = pd.DataFrame({"foo": [1, 2, 3], "bar": [4, 5, 6]})

In [25]: df = df.reset_index(drop=True)

In [26]: df.iloc[0, 0] = 100

In [27]: df
Out[27]: 
   foo  bar
0  100    4
1    2    5
2    3    6

The initial object gets out of scope as soon as the result of is reassigned and hence does not share data with any other object. No copy is necessary when modifying the object. This is generally true for all methods listed in Copy-on-Write optimizations.reset_indexdf

In [2]: subset = df["foo"]

Chained Assignment#

Chained assignment references a technique where an object is updated through two subsequent indexing operations, e.g.

In [1]: df = pd.DataFrame({"foo": [1, 2, 3], "bar": [4, 5, 6]})
In [2]: df["foo"][df["bar"] > 5] = 100
In [3]: df
Out[3]:
   foo  bar
0    1    4
1    2    5
2  100    6

The column was updated where the column is greater than 5. This violated the CoW principles though, because it would have to modify the view and in one step. Hence, chained assignment will consistently never work and raise a warning with CoW enabled:foobardf["foo"]dfChainedAssignmentError

In [33]: df = pd.DataFrame({"foo": [1, 2, 3], "bar": [4, 5, 6]})

In [34]: df["foo"][df["bar"] > 5] = 100

With copy on write this can be done by using .loc

In [35]: df.loc[df["bar"] > 5, "foo"] = 100

Read-only NumPy arrays#

Accessing the underlying NumPy array of a DataFrame will return a read-only array if the array shares data with the initial DataFrame:

The array is a copy if the initial DataFrame consists of more than one array:

In [36]: df = pd.DataFrame({"a": [1, 2], "b": [1.5, 2.5]})

In [37]: df.to_numpy()
Out[37]: 
array([[1. , 1.5],
       [2. , 2.5]])

The array shares data with the DataFrame if the DataFrame consists of only one NumPy array:

In [38]: df = pd.DataFrame({"a": [1, 2], "b": [3, 4]})

In [39]: df.to_numpy()
Out[39]: 
array([[1, 3],
       [2, 4]])

This array is read-only, which means that it can’t be modified inplace:

In [40]: arr = df.to_numpy()

In [41]: arr[0, 0] = 100
---------------------------------------------------------------------------
ValueError                                Traceback (most recent call last)
Cell In[41], line 1
----> 1 arr[0, 0] = 100

ValueError: assignment destination is read-only

The same holds true for a Series, since a Series always consists of a single array.

There are two potential solutions to this:

  • Trigger a copy manually if you want to avoid updating DataFrames that share memory with your array.

  • Make the array writeable. This is a more performant solution but circumvents Copy-on-Write rules, so it should be used with caution.

In [42]: arr = df.to_numpy()

In [43]: arr.flags.writeable = True

In [44]: arr[0, 0] = 100

In [45]: arr
Out[45]: 
array([[100,   3],
       [  2,   4]])

Patterns to avoid#

No defensive copy will be performed if two objects share the same data while you are modifying one object inplace.

In [46]: df = pd.DataFrame({"a": [1, 2, 3], "b": [4, 5, 6]})

In [47]: df2 = df.reset_index(drop=True)

In [48]: df2.iloc[0, 0] = 100

This creates two objects that share data and thus the setitem operation will trigger a copy. This is not necessary if the initial object isn’t needed anymore. Simply reassigning to the same variable will invalidate the reference that is held by the object.df

In [49]: df = pd.DataFrame({"a": [1, 2, 3], "b": [4, 5, 6]})

In [50]: df = df.reset_index(drop=True)

In [51]: df.iloc[0, 0] = 100

No copy is necessary in this example. Creating multiple references keeps unnecessary references alive and thus will hurt performance with Copy-on-Write.

Copy-on-Write optimizations#

A new lazy copy mechanism that defers the copy until the object in question is modified and only if this object shares data with another object. This mechanism was added to methods that don’t require a copy of the underlying data. Popular examples are DataFrame.drop() for and DataFrame.rename().axis=1

These methods return views when Copy-on-Write is enabled, which provides a significant performance improvement compared to the regular execution.
