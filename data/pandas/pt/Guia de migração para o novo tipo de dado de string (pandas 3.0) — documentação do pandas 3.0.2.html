Guia de migração para o novo tipo de dado de string (pandas 3.0)#

O próximo lançamento do pandas 3.0 introduz um novo tipo de dado padrão de string. Isso Provavelmente vai causar algum trabalho ao atualizar para o Pandas 3.0, e esta página Fornece uma visão geral dos problemas que você pode encontrar e orienta como para enfrentá-los.

Esse novo dtype já está disponível na versão 2.3 do pandas, e você pode Habilite-o com:

pd.options.future.infer_string = True

Isso permite que você teste seu código antes do lançamento final da 3.0.

Nota

Este guia de migração foca nas mudanças e etapas necessárias quando Você está atualmente usando dType para dados de string, que é usado por Padrão no Pandas < 3.0. Se você já estiver usando uma das cadeias de opt-in dtypes, você pode continuar fazendo isso sem mudanças. Veja Para usuários existentes do StringDtype anulável para mais detalhes.object

Contexto#

Historicamente, pandas sempre usou o dtype NumPy como padrão para armazenar dados de texto. Isso tem duas principais desvantagens. Primeiro, dtype é não específico para strings: qualquer objeto Python pode ser armazenado em um -dtype array, não apenas strings, e considerando que o dtype para uma coluna com Strings é confuso para os usuários. Segundo, isso nem sempre é muito eficiente (ambos em termos de desempenho e para uso de memória).objectobjectobjectobject

Desde o pandas 1.0, um tipo de dado de string opt-in está disponível, mas isso tem sido ainda não foi tornado padrão, e usa o escalar para representar Valores faltando.pd.NA

Pandas 3.0 altera o dtype padrão para strings para um novo tipo de dado de string, uma variante do tipo de dado de string opcional existente, mas usando como o indicador de valor ausente, para ser consistente com os outros tipos de dados padrão.NaN

Para melhorar o desempenho, o novo tipo de dado de string usará o pacote por padrão, se instalado (e caso contrário, usa object dtype sob o capuz como plano B).pyarrow

Veja PDEP-14: Tipo de dado dedicado de string para pandas 3.0 para mais informações e contexto.

Breve introdução ao novo tipo padrão de cadeias dtype#

Por padrão, pandas inferirá essa nova string dtype em vez de object dtype para dados de string (ao criar objetos pandas, como em construtores ou IO funções).

Ser um dtype padrão significa que o dtype da string será usado em métodos de IO ou construtores quando o dtype está sendo inferido e a entrada é inferida como sendo Dados da cadeia:

>>> pd.Series(["a", "b", None])
0      a
1      b
2    NaN
dtype: str

Também pode ser especificado explicitamente usando o alias:"str"

>>> pd.Series(["a", "b", None], dtype="str")
0      a
1      b
2    NaN
dtype: str

Similarly, functions like read_csv(), read_parquet(), and others will now use the new string dtype when reading string data.

In contrast to the current object dtype, the new string dtype will only store strings. This also means that it will raise an error if you try to store a non-string value in it (see below for more details).

Missing values with the new string dtype are always represented as (), and the missing value behavior is similar to other default dtypes.NaNnp.nan

This new string dtype should otherwise behave the same as the existing dtype users are used to. For example, all string-specific methods through the accessor will work the same:objectstr

>>> ser = pd.Series(["a", "b", None], dtype="str")
>>> ser.str.upper()
0    A
1    B
2  NaN
dtype: str

Note

The new default string dtype is an instance of the pandas.StringDtype class. The dtype can be constructed as , but for general usage we recommend to use the shorter alias.pd.StringDtype(na_value=np.nan)"str"

Overview of behavior differences and how to address them#

The dtype is no longer a numpy “object” dtype#

When inferring or reading string data, the data type of the resulting DataFrame column or Series will silently start being the new dtype instead of the numpy dtype, and this can have some impact on your code."str""object"

The new string dtype is a pandas data type (“extension dtype”), and no longer a numpy instance. Therefore, passing the dtype of a string column to numpy functions will no longer work (e.g. passing it to a argument of a numpy function, or using to check the dtype).np.dtypedtype=np.issubdtype

Checking the dtype#

When checking the dtype, code might currently do something like:

>>> ser = pd.Series(["a", "b", "c"])
>>> ser.dtype == "object"

to check for columns with string data (by checking for the dtype being ). This will no longer work in pandas 3+, since will now be with the new default string dtype, and the above check will return ."object"ser.dtype"str"False

To check for columns with string data, you should instead use:

>>> ser.dtype == "str"

How to write compatible code

For code that should work on both pandas 2.x and 3.x, you can use the pandas.api.types.is_string_dtype() function:

>>> pd.api.types.is_string_dtype(ser.dtype)
True

This will return for both the object dtype and the string dtypes.True

Hardcoded use of object dtype#

If you have code where the dtype is hardcoded in constructors, like

>>> pd.Series(["a", "b", "c"], dtype="object")

this will keep using the object dtype. You will want to update this code to ensure you get the benefits of the new string dtype.

How to write compatible code?

First, in many cases it can be sufficient to remove the specific data type, and let pandas do the inference. But if you want to be specific, you can specify the dtype:"str"

>>> pd.Series(["a", "b", "c"], dtype="str")

This is actually compatible with pandas 2.x as well, since in pandas < 3, was essentially treated as an alias for object dtype.dtype="str"

Attention

While using in constructors is compatible with pandas 2.x, specifying it as the dtype in astype() runs into the issue of also stringifying missing values in pandas 2.x. See the section astype(str) preserving missing values for more details.dtype="str"

For selecting string columns with select_dtypes() in a pandas 2.x and 3.x compatible way, it is not possible to use . While this works for pandas 3.x, it raises an error in pandas 2.x. As an alternative, you can select both (for pandas 2.x) and (for pandas 3.x; which will also select the default dtype and does not error on pandas 2.x):"str"object"string"str

# can use ``include=["str"]`` for pandas >= 3
>>> df.select_dtypes(include=["object", "string"])

The missing value sentinel is now always NaN#

When using object dtype, multiple possible missing value sentinels are supported, including and . With the new default string dtype, the missing value sentinel is always NaN ():Nonenp.nannp.nan

# with object dtype, None is preserved as None and seen as missing
>>> ser = pd.Series(["a", "b", None], dtype="object")
>>> ser
0       a
1       b
2    None
dtype: object
>>> print(ser[2])
None

# with the new string dtype, any missing value like None is coerced to NaN
>>> ser = pd.Series(["a", "b", None], dtype="str")
>>> ser
0      a
1      b
2    NaN
dtype: str
>>> print(ser[2])
nan

Generally this should be no problem when relying on missing value behavior in pandas methods (for example, will give the same result as before). But when you relied on the exact value of being present, that can impact your code.ser.isna()None

How to write compatible code?

When checking for a missing value, instead of checking for the exact value of or , you should use the pandas.isna() function. This is the most robust way to check for missing values, as it will work regardless of the dtype and the exact missing value sentinel:Nonenp.nan

>>> pd.isna(ser[2])
True

One caveat: this function works both on scalars and on array-likes, and in the latter case it will return an array of bools. When using it in a Boolean context (for example, ) be sure to only pass a scalar to it.if pd.isna(..): ..

“setitem” operations will now raise an error for non-string data#

With the new string dtype, any attempt to set a non-string value in a Series or DataFrame will raise an error:

>>> ser = pd.Series(["a", "b", None], dtype="str")
>>> ser[1] = 2.5
---------------------------------------------------------------------------
TypeError                                 Traceback (most recent call last)
...
TypeError: Invalid value '2.5' for dtype 'str'. Value should be a string or missing value, got 'float' instead.

If you relied on the flexible nature of object dtype being able to hold any Python object, but your initial data was inferred as strings, your code might be impacted by this change.

How to write compatible code?

You can update your code to ensure you only set string values in such columns, or otherwise you can explicitly ensure the column has object dtype first. This can be done by specifying the dtype explicitly in the constructor, or by using the astype() method:

>>> ser = pd.Series(["a", "b", None], dtype="str")
>>> ser = ser.astype("object")
>>> ser[1] = 2.5

This call will be redundant when using pandas 2.x, but this code will work for all versions.astype("object")

Invalid unicode input#

Python allows to have a built-in object that represents invalid unicode data. And since the dtype can hold any Python object, you can have a pandas Series with such invalid unicode data:strobject

>>> ser = pd.Series(["\u2600", "\ud83d"], dtype=object)
>>> ser
0    ☀
1    \ud83d
dtype: object

However, when using the string dtype using under the hood, this can only store valid unicode data, and otherwise it will raise an error:pyarrow

>>> ser = pd.Series(["\u2600", "\ud83d"])
---------------------------------------------------------------------------
UnicodeEncodeError                        Traceback (most recent call last)
...
UnicodeEncodeError: 'utf-8' codec can't encode character '\ud83d' in position 0: surrogates not allowed

If you want to keep the previous behaviour, you can explicitly specify to keep working with object dtype.dtype=object

When you have byte data that you want to convert to strings using , the decode() method now has a parameter to be able to specify object dtype instead of the default of string dtype for this use case.decode()dtype

Series.values() now returns an ExtensionArray#

With object dtype, using on a Series will return the underlying NumPy array..values

>>> ser = pd.Series(["a", "b", np.nan], dtype="object")
>>> type(ser.values)
<class 'numpy.ndarray'>

However with the new string dtype, the underlying ExtensionArray is returned instead.

>>> ser = pd.Series(["a", "b", pd.NA], dtype="str")
>>> ser.values
<ArrowStringArray>
['a', 'b', nan]
Length: 3, dtype: str

If your code requires a NumPy array, you should use Series.to_numpy().

>>> ser = pd.Series(["a", "b", pd.NA], dtype="str")
>>> ser.to_numpy()
['a' 'b' nan]

In general, you should always prefer Series.to_numpy() to get a NumPy array or Series.array() to get an ExtensionArray over using Series.values().

Notable bug fixes#

astype(str) preserving missing values#

The stringifying of missing values is a long standing “bug” or misfeature, as discussed in pandas-dev/pandas#25353, but fixing it introduces a significant behaviour change.

With pandas < 3, when using or , the operation would convert every element to a string, including the missing values:astype(str)astype("str")

# OLD behavior in pandas < 3
>>> ser = pd.Series([1.5, np.nan])
>>> ser
0    1.5
1    NaN
dtype: float64
>>> ser.astype("str")
0    1.5
1    nan
dtype: object
>>> ser.astype("str").to_numpy()
array(['1.5', 'nan'], dtype=object)

Note how () was converted to the string . This was not the intended behavior, and it was inconsistent with how other dtypes handled missing values.NaNnp.nan"nan"

With pandas 3, this behavior has been fixed, and now will cast to the new string dtype, which preserves the missing values:astype("str")

# NEW behavior in pandas 3
>>> pd.options.future.infer_string = True
>>> ser = pd.Series([1.5, np.nan])
>>> ser.astype("str")
0    1.5
1    NaN
dtype: str
>>> ser.astype("str").to_numpy()
array(['1.5', nan], dtype=object)

If you want to preserve the old behaviour of converting every object to a string, you can use instead. If you want do such conversion while preserving the missing values in a way that works with both pandas 2.x and 3.x, you can use (for pandas 3.x only, you can do ).ser.map(str)ser.map(str, na_action="ignore")ser.astype("str")

If you want to convert to object or string dtype for pandas 2.x and 3.x, respectively, without needing to stringify each individual element, you will have to use a conditional check on the pandas version. For example, to convert a categorical Series with string categories to its dense non-categorical version with object or string dtype:

>>> import pandas as pd
>>> ser = pd.Series(["a", np.nan], dtype="category")
>>> ser.astype(object if pd.__version__ < "3" else "str")

prod() raising for string data#

In pandas < 3, calling the prod() method on a Series with string data would generally raise an error, except when the Series was empty or contained only a single string (potentially with missing values):

>>> ser = pd.Series(["a", None], dtype=object)
>>> ser.prod()
'a'

When the Series contains multiple strings, it will raise a . This behaviour stays the same in pandas 3 when using the flexible dtype. But by virtue of using the new string dtype, this will generally consistently raise an error regardless of the number of strings:TypeErrorobject

>>> ser = pd.Series(["a", None], dtype="str")
>>> ser.prod()
---------------------------------------------------------------------------
TypeError                                 Traceback (most recent call last)
...
TypeError: Cannot perform reduction 'prod' with string dtype
