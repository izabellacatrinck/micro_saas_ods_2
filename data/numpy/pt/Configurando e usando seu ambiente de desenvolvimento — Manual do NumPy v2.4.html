Configurando e utilizando seu ambiente de desenvolvimento#

Configuração recomendada de desenvolvimento#

Como o NumPy contém partes escritas em C e Cython, isso precisa ser compilado antes de usar, certifique-se de ter os compiladores necessários e Python cabeçalhos de desenvolvimento instalados - veja Construindo a partir da fonte. Construção O NumPy na versão requer compiladores compatíveis com C11 e C++17.2.0

Ter código compilado também significa importar o NumPy do desenvolvimento Fontes precisam de algumas etapas adicionais, que são explicadas abaixo. Para o resto Neste capítulo, assumimos que você configurou seu repositório git conforme descrito em Trabalhar com código-fonte scikit-image.

Nota

Se você está tendo dificuldades para montar o NumPy a partir da fonte ou configurar seu ambiente de desenvolvimento local, você pode tentar construir o NumPy com o GitHub Codespaces. Isso permite criar o ambiente de desenvolvimento correto Diretamente no seu navegador, reduzindo a necessidade de instalar desenvolvimento local e lidam com dependências incompatíveis.

Se você tem boa conectividade à internet e quer uma configuração temporária, é isso muitas vezes mais rápido para trabalhar no NumPy em um ambiente Codespaces. Para documentação sobre como começar com o Codespaces, veja a documentação do Codespaces. Ao criar um espaço de código para o repositório, o padrão Funciona do tipo máquina de 2 núcleos; O 4-core vai construir e funcionar um pouco mais rápido (mas de ao custo de reduzir pela metade seu número de horas de uso gratuito). Quando seu O espaço de código começou, você pode rodar e seu O ambiente de desenvolvimento está completamente configurado - você pode então seguir o partes relevantes da documentação do NumPy para construir, testar, desenvolver e escrever documentação, e contribuir para o NumPy.numpy/numpyconda activate numpy-dev

Uso de ambientes virtuais#

Uma pergunta frequente é: "Como configuro uma versão de desenvolvimento do NumPy em paralelo a uma versão lançada que uso para fazer meu trabalho/pesquisa?".

Uma maneira simples de conseguir isso é instalar a versão lançada em site-packages, usando pip ou conda por exemplo, e set Atualize a versão de desenvolvimento em um ambiente virtual.

Se você usa o Conda, recomendamos criar um ambiente virtual separado para Desenvolvimento numpy usando o arquivo na raiz do repositório (isso criará o ambiente e instalará todas as dependências de desenvolvimento em uma vez):environment.yml

$ conda env create -f environment.yml  # `mamba` works too for this command
$ conda activate numpy-dev

Se você instalou Python de outra forma que não seja o conda, primeiro instale o virtualenv (opcionalmente use virtualenvwrapper), depois crie seu virtualenv (nomeado aqui), ative e instale todos os projetos Dependências com:numpy-dev

$ virtualenv numpy-dev
$ source numpy-dev/bin/activate # activate virtual environment
$ python -m pip install -r requirements/all_requirements.txt

Agora, sempre que você quiser mudar para o ambiente virtual, pode usar o comando , e para sair do Ambiente virtual e de volta à sua concha anterior.source numpy-dev/bin/activatedeactivate

Testing builds#

Before running the tests, first install the test dependencies:

$ python -m pip install -r requirements/test_requirements.txt
$ python -m pip install asv # only for running benchmarks

To build the development version of NumPy and run tests, spawn interactive shells with the Python import paths properly set up etc., use the spin utility. To run tests, do one of:

$ spin test -v
$ spin test numpy/random  # to run the tests in a specific module
$ spin test -v -t numpy/_core/tests/test_nditer.py::test_iter_c_order
$ spin test -p auto # to run tests in parallel threads using pytest-run-parallel

This builds NumPy first, so the first time it may take a few minutes.

You can also use for benchmarking. See for more command line options.spin benchspin --help

Note

If the above commands result in , run .RuntimeError: Cannot parse version 0+untagged.xxxxxgit pull upstream main --tags

Additional arguments may be forwarded to by passing the extra arguments after a bare . For example, to run a test method with the flag forwarded to the target, run the following:pytest----pdb

$ spin test -t numpy/tests/test_scripts.py::test_f2py -- --pdb

You can also match test names using python operators by passing the argument to pytest:-k

$ spin test -v -t numpy/_core/tests/test_multiarray.py -- -k "MatMul and not vector"

To run “doctests” – to check that the code examples in the documentation is correct – use the check-docs spin command. It relies on the scipy-docs package, which provides several additional features on top of the standard library package. Install and run one of:doctestscipy-doctest

$ spin check-docs -v
$ spin check-docs numpy/linalg
$ spin check-docs -v -- -k 'det and not slogdet'

Note

Remember that all tests of NumPy should pass before committing your changes.

Note

Some of the tests in the test suite require a large amount of memory, and are skipped if your system does not have enough.

Running tests#

Besides using , there are various ways to run the tests. Inside the interpreter, tests can be run like this:spin

>>> np.test()  
>>> np.test('full')   # Also run tests marked as slow
>>> np.test('full', verbose=2)   # Additionally print test name/file

An example of a successful test :
``4686 passed, 362 skipped, 9 xfailed, 5 warnings in 213.99 seconds``

Or a similar way from the command line:

$ python -c "import numpy as np; np.test()"

Tests can also be run with , however then the NumPy-specific plugin is not found which causes strange side effects.pytest numpy

Running individual test files can be useful; it’s much faster than running the whole test suite or that of a whole module (example: ). This can be done with:np.random.test()

$ python path_to_testfile/test_file.py

That also takes extra arguments, like which drops you into the Python debugger when a test fails or an exception is raised.--pdb

Running tests with tox is also supported. For example, to build NumPy and run the test suite with Python 3.9, use:

$ tox -e py39

For more extensive information, see Testing guidelines.

Note: do not run the tests from the root directory of your numpy git repo without , that will result in strange test errors.spin

Changes that involve static type declarations are also executed using . The invocation will look like the following:spin

$ spin mypy

This will look in the directory for sets of operations to test for type incompatibility.typing/tests

Running linting#

Lint checks can be performed on newly added lines of Python code.

Install all dependent packages using pip:

$ python -m pip install -r requirements/linter_requirements.txt

To run lint checks before committing new code, run:

$ python tools/linter.py

To check all changes in newly added Python code of current branch with target branch, run:

$ python tools/linter.py

If there are no errors, the script exits with no message. In case of errors, check the error message for details:

$ python tools/linter.py
./numpy/_core/tests/test_scalarmath.py:34:5: E303 too many blank lines (3)
1       E303 too many blank lines (3)

It is advisable to run lint checks before pushing commits to a remote branch since the linter runs as part of the CI pipeline.

For more details on Style Guidelines:

  • Python Style Guide

  • NEP 45 — C style guide

Rebuilding & cleaning the workspace#

Rebuilding NumPy after making changes to compiled code can be done with the same build command as you used previously - only the changed files will be re-built. Doing a full build, which sometimes is necessary, requires cleaning the workspace first. The standard way of doing this is (note: deletes any uncommitted files!):

$ git clean -xdf

When you want to discard all changes and go back to the last commit in the repo, use one of:

$ git checkout .
$ git reset --hard

Debugging#

Another frequently asked question is “How do I debug C code inside NumPy?”. First, ensure that you have gdb installed on your system with the Python extensions (often the default on Linux). You can see which version of Python is running inside gdb to verify your setup:

(gdb) python
>import sys
>print(sys.version_info)
>end
sys.version_info(major=3, minor=7, micro=0, releaselevel='final', serial=0)

Most python builds do not include debug symbols and are built with compiler optimizations enabled. To get the best debugging experience using a debug build of Python is encouraged, see Advanced debugging tools.

In terms of debugging, NumPy also needs to be built in a debug mode. You need to use build type and disable optimizations to make sure flag is used during object building. Note that NumPy should NOT be installed in your environment before you build with the command.debug-O0spin build

To generate source-level debug information within the build process run:

$ spin build --clean -- -Dbuildtype=debug -Ddisable-optimization=true

Next you need to write a Python script that invokes the C code whose execution you want to debug. For instance :mytest.py

import numpy as np
x = np.arange(5)
np.empty_like(x)

Note that your test file needs to be outside the NumPy clone you have. Now, you can run:

$ spin gdb /path/to/mytest.py

In case you are using clang toolchain:

$ spin lldb /path/to/mytest.py

And then in the debugger:

(gdb) break array_empty_like
(gdb) run

lldb counterpart:

(lldb) breakpoint set --name array_empty_like
(lldb) run

The execution will now stop at the corresponding C function and you can step through it as usual. A number of useful Python-specific commands are available.   • next: Step program, proceeding through subroutine calls.

  • step: Continue program being debugged, after signal or breakpoint.

  • print: Print value of expression EXP.

Rich support for Python debugging requires that the script distributed with Python is installed in a path where gdb can find it. If you installed your Python build from your system package manager, you likely do not need to manually do anything. However, if you built Python from source, you will likely need to create a file in your home directory pointing gdb at the location of your Python installation. For example, a version of python installed via pyenv needs a file with the following contents:python-gdb.py.gdbinit.gdbinit

Understanding the code & getting started#

The best strategy to better understand the code base is to pick something you want to change and start reading the code to figure out how it works. When in doubt, you can ask questions on the mailing list. It is perfectly okay if your pull requests aren’t perfect, the community is always happy to help. As a volunteer project, things do sometimes get dropped and it’s totally fine to ping us if something has sat without a response for about two to four weeks.

So go ahead and pick something that annoys or confuses you about NumPy, experiment with the code, hang around for discussions or go through the reference documents to try to fix it. Things will fall in place and soon you’ll have a pretty good understanding of the project as a whole. Good Luck!
