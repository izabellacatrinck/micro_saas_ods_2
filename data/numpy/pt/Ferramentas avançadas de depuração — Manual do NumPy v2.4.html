Ferramentas avançadas de depuração#

Se você chegou aqui, quer mergulhar ou usar ferramentas mais avançadas. Isso geralmente não é necessário para colaboradores iniciantes e para a maioria dos colaboradores desenvolvimento do dia a dia. Esses são usados com mais raridade, por exemplo, próximos a um novo lançamento do NumPy, ou quando uma mudança grande ou complexa era feita.

Algumas dessas ferramentas são usadas nos testes de integração contínua do NumPy. Se você Veja uma falha de teste que só ocorre em uma ferramenta de depuração, estas instruções Espero que você possa reproduzir a falha do teste localmente.

Como nem todas essas ferramentas são usadas regularmente e estão disponíveis apenas Em alguns sistemas, espere diferenças, problemas ou peculiaridades; Ficaremos felizes em ajudar caso você trave e apreciemos qualquer melhoria ou sugestões para esses fluxos de trabalho.

Encontrando erros em C com ferramentas adicionais#

A maioria dos desenvolvimentos não exigirá mais do que uma cadeia típica de ferramentas de depuração como mostrado em Depuração. Mas, por exemplo, vazamentos de memória podem ser particularmente sutis ou difíceis de Reduza as opções.

Não esperamos que nenhuma dessas ferramentas seja executada pela maioria dos colaboradores. No entanto, você pode garantir que possamos identificar esses problemas com mais facilidade:

  • Os testes devem cobrir todos os caminhos de código, incluindo caminhos de erro.

  • Tente escrever testes curtos e simples. Se você tem um teste muito complicado Considere criar um teste mais simples também. Isso pode ser útil, porque muitas vezes é fácil encontrar qual teste Desencadeia um problema e não qual linha do teste.

  • Nunca use se os dados forem lidos/usados. Valgrind vai perceber isso e reporte um erro. Quando você não se importa com valores, pode gerar valores aleatórios em vez disso.np.empty

Isso nos ajudará a identificar qualquer falha antes que sua alteração seja divulgada e significa que você não precisa se preocupar em cometer erros de contagem de referências, O que pode ser intimidador.

Build de depuração em Python#

Compilações de depuração em Python estão facilmente disponíveis, por exemplo, pelo pacote do sistema gerenciador em sistemas Linux, mas também estão disponíveis em outras plataformas, possivelmente em um formato menos conveniente. Se você não conseguir instalar facilmente uma build de depuração em Python A partir de um gerenciador de pacotes de sistema, você pode construir um você mesmo usando pyenv. valgrind#

Valgrind is a powerful tool to find certain memory access problems and should be run on complicated C code. Basic use of usually requires no more than:valgrind

PYTHONMALLOC=malloc valgrind python runtests.py

where is necessary to avoid false positives from python itself. Depending on the system and valgrind version, you may see more false positives. supports “suppressions” to ignore some of these, and Python does have a suppression file (and even a compile time option) which may help if you find it necessary.PYTHONMALLOC=mallocvalgrind

Valgrind helps:

  • Find use of uninitialized variables/memory.

  • Detect memory access violations (reading or writing outside of allocated memory).

  • Find many memory leaks. Note that for most leaks the python debug build approach (and ) is much more sensitive. The reason is that can only detect if memory is definitely lost. If:pytest-leaksvalgrind

    dtype = np.dtype(np.int64)
arr.astype(dtype=dtype)

    Has incorrect reference counting for , this is a bug, but valgrind cannot see it because always returns the same object. However, not all dtypes are singletons, so this might leak memory for different input. In rare cases NumPy uses and not the Python memory allocators which are invisible to the Python debug build. should normally be avoided, but there are some exceptions (e.g. the structure is public API and cannot use the Python allocators.)dtypenp.dtype(np.int64)mallocmallocPyArray_Dims

Even though using valgrind for memory leak detection is slow and less sensitive it can be convenient: you can run most programs with valgrind without modification.

Things to be aware of:

  • Valgrind does not support the numpy , this means that tests will fail or be flagged errors that are completely fine.longdouble

  • Expect some errors before and after running your NumPy code.

  • Caches can mean that errors (specifically memory leaks) may not be detected or are only detect at a later, unrelated time.

A big advantage of valgrind is that it has no requirements aside from valgrind itself (although you probably want to use debug builds for better tracebacks).

C debuggers#

Whenever NumPy crashes or when working on changes to NumPy’s low-level C or C++ code, it’s often convenient to run Python under a C debugger to get more information. A debugger can aid in understanding an interpreter crash (e.g. due to a segmentation fault) by providing a C call stack at the site of the crash. The call stack often provides valuable context to understand the nature of a crash. C debuggers are also very useful during development, allowing interactive debugging in the C implementation of NumPy.

The NumPy developers often use both and to debug Numpy. As a rule of thumb, is often easier to use on Linux while is easier to use on a Mac environment. They have disjoint user interfaces, so you will need to learn how to use whichever one you land on. The to command map is a convenient reference for how to accomplish common recipes in both debuggers.gdblldbgdblldbgdblldb

Running a Test Script#

Let’s say you have a test script named test.py that lives in a folder in the same directory as the NumPy source checkout. You could execute the test script using the build of NumPy with the following incantation:testspin

spin gdb ../test/test.py

This will launch into gdb. If all you care about is a call stack for a crash, type “r” and hit enter. Your test script will run and if a crash happens, you type “bt” to get a traceback. For , the instructions are similar, just replace with .lldbspin gdbspin lldb

You can also set breakpoints and use other more advanced techniques. See the documentation for your debugger for more details.

One common issue with breakpoints in NumPy is that some code paths get hit repeatedly during the import of the module. This can make it tricky or tedious to find the first “real” call after the NumPy import has completed and the module is fully initialized.numpynumpy

One workaround is to use a script like this:

import os
import signal

import numpy as np

PID = os.getpid()

def do_nothing(*args):
    pass

signal.signal(signal.SIGUSR1, do_nothing)

os.kill(PID, signal.SIGUSR1)

# the code to run under a debugger follows

This example installs a signal handler for the signal that does nothing and then calls on the Python process with the signal. This causes the signal handler to fire and critically also causes both and to halt execution inside of the syscall.SIGUSR1os.killSIGUSR1gdblldbkill

Compiler Sanitizers#

The compiler sanitizer suites shipped by both GCC and LLVM offer a means to detect many common programming errors at runtime. The sanitizers work by instrumenting the application code at build time so additional runtime checks fire. Typically, sanitizers are run during the course of regular testing and if a sanitizer check fails, this leads to a test failure or crash, along with a report about the nature of the failure.

While it is possible to use sanitizers with a “regular” build of CPython - it is best if you can set up a Python environment based on a from-source Python build with sanitizer instrumentation, and then use the instrumented Python to build NumPy and run the tests. If the entire Python stack is instrumented using the same sanitizer runtime, it becomes possible to identify issues that happen across the Python stack. This enables detecting memory leaks in NumPy due to misuse of memory allocated in CPython, for example.

Build Python with Sanitizer Instrumentation#

See the section in the Python developer’s guide on this topic for more information about building Python from source. To enable address sanitizer, you will need to pass to the script invocation when you build Python.--with-address-sanitizerconfigure

You can also use pyenv to automate the process of building Python and quickly activate or deactivate a Python installation using a command-line interface similar to virtual environments. With you could install an ASAN-instrumented build of Python 3.13 like this:pyenv

CONFIGURE_OPTS="--with-address-sanitizer" pyenv install 3.13

If you are interested in thread sanitizer, the docker images might also be a quicker choice that bypasses building Python from source, although it may be annoying to do debugging work inside of a docker image.cpython_sanity

Use together with spin#

However you build Python, once you have an instrumented Python build, you can install NumPy’s development and test dependencies and build NumPy with address sanitizer instrumentation. For example, to build NumPy with the profile and address sanitizer, you would pass additional build options to like this:debugmeson

spin build -- -Dbuildtype=debug -Db_sanitize=address

Once the build is finished, you can use other command like and as with any other Python build.spinspin testspin gdb

Special considerations#

Some NumPy tests intentionally lead to returning . In its default configuration, some of the compiler sanitizers flag this as an error. You can disable that check by passing to the sanitizer as an option. For example, with address sanitizer:mallocNULLallocator_may_return_null=1

ASAN_OPTIONS=allocator_may_return_null=1 spin test

You may see memory leaks coming from the Python interpreter, particularly on MacOS. If the memory leak reports are not useful, you can disable leak detection by passing in . You can pass more than one option using a colon-delimited list, like this:detect_leaks=0ASAN_OPTIONS

ASAN_OPTIONS=allocator_may_return_null=1:halt_on_error=1:detect_leaks=1 spin test

The option can be particularly useful – it hard-crashes the Python executable whenever it detects an error, along with a report about the error that includes a stack trace.halt_on_error
